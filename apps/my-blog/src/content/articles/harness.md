## 开篇：这篇文章想解决什么？

最近关于 harness、harness engineering、managed agents 的文章越来越多。之所以越看越乱，不是因为这些文章彼此冲突，而是因为它们都在用 "harness" 这个词，但讨论的层面并不一样——有些文章关注模型本体怎么调用工具、维护状态、长时间执行任务；另一些文章关注代码库、文档、测试、CI、可观测性和团队流程怎么围绕 Agent 重构。

更值得警惕的是，同一时期还有一股 "AI-First" 的叙事——很多团队宣称自己是 AI-First，但实际做法只是"给工程师装了个 Cursor、给 PM 装了个 ChatGPT"。Peter Pang（CreaoAI 联合创始人，前 Meta GenAI / LLaMA）在 [Why Your "AI-First" Strategy Is Probably Wrong](https://x.com/intuitiveml/article/2043545596699750791) 里把这种状态命名得很精确：

> Most companies bolt AI onto their existing process … Efficiency goes up 10 to 20 percent. Nothing structurally changes. **That is AI-assisted.**
> AI-first means you redesign your process, your architecture, and your organization around the assumption that **AI is the primary builder**.
> Pang 在 CREAO 用两个月做的事情，正是这种结构性重设的一个具体案例：不是在原有流程上“外挂 AI”，而是反过来围绕 AI 重新组织研发流程、系统边界和交付方式。更关键的是，Pang 对这件事的总结与 OpenAI 后来提出的 harness engineering 几乎同构。

更关键的是 Pang 自己的总结：

> OpenAI published a concept in February 2026 that captured what we'd been doing. They called it harness engineering … **We arrived at that conclusion on our own.** We didn't have a name for it.
> 这是一线 CTO 对 OpenAI 那套 harness engineering 论述的独立印证——同一个问题，从不同方向走过去会走到同一个结论：**真正的 AI-First / Harness Engineering 不是"多用一个模型"，而是"把研发系统改造成对 Agent 友好的工程环境"。Pang 用一句话概括了它的代价：**

> **Fast AI without fast validation is fast-moving technical debt.**

##### 这篇文章想做两件事：

1. 先把 "harness" 这个术语拆清楚，让团队有一个共同坐标系，知道我们在谈哪一层；
2. 再沿着这个坐标系落到工程实施——当研发系统开始围绕 Agent 重构时，我们到底应该建设什么。

写作起点是 OpenAI 2026 年 2 月那篇 [Harness engineering: leveraging Codex in an agent-first world](https://openai.com/index/harness-engineering/)。它把讨论从"Agent 本体怎么运行"明确推进到了"工程环境怎么为了 agent-first 开发而重构"。下文的框架，可以理解为对那篇文章的陪读与延伸，同时也是对 Peter Pang 那个问题——"How do we restructure everything so AI does the building, and engineers provide direction and judgment?"——的一个正面回答。

## 一、先建立坐标系

### （一）两个层面的 Harness

一个更实用的做法不是追求一个覆盖所有语境的统一定义，而是先把问题分成两类：

- **Agent Harness**：模型之外，让 Agent 能感知、规划、调用工具、执行动作、接收反馈并持续完成任务的运行与控制系统。它解决的是：模型怎么访问工具和环境、怎么持久化状态跨会话继续工作、怎么控制上下文膨胀、怎么做规划执行验证和重试、多个 Agent 怎么协作分工。如果借用 LangChain 的表达，可以写成 `Agent = Model + Harness`。
- **面向 Agent 软件工程的 Harness**：围绕 Agent 重构代码库、文档、测试、可观测性和开发流程，使 Agent 能在真实软件工程里稳定协作、验证和交付的外层系统。它解决的是：repo 是否对 Agent 可读、docs 和规则是否结构化、tests / CI / review / 验收是否能进入 agent loop、UI 和日志指标 trace 是否对 Agent 可见、团队流程是否支持 agent-first 的开发方式。

一句更直白的话：

> Agent Harness 让 Agent **跑起来**；面向 Agent 软件工程的 Harness 让 Agent 在真实工程里**产生价值**。
> Peter Pang 在 CREAO 用两个月落地的整套系统（Monorepo 让 AI 看见全貌、六阶段确定性 CI/CD、AI 代码审查、自动分诊与自愈循环），以及 OpenAI 团队五个月里建起来的那套（AGENTS.md 作为目录、可观测性对 Agent 可消费、doc-gardening、垃圾回收式重构），**全部都落在第二层**。两条独立路径都在印证同一件事：

> 对团队落地来说，**工程环境层的 Harness 比 Agent 本体的 Harness 重要得多**。模型能力已经足够好，真正卡住落地的，是我们自己的工程系统还没准备好。
> 需要说明：这个二分不是各篇原文统一采用的标准术语，而是本文为了组织讨论使用的工作框架。不同公司在用 harness 这个词时侧重点各不相同，这里只是把它们放进一个更容易理解的位置。

## 二、定调与总纲

### （一）这不是"多用一个模型"，而是"让研发系统开始对 Agent 友好"

很多人第一次接触这个话题时，容易把它理解成：

- 我们是不是多用了一个更强的模型？
- 是不是把 Copilot 换成了 Claude Code、Codex 这类更强的 coding agent？
- 是不是给工程师加了一层自动化工具？

这正是 Peter Pang 所说的 **AI-assisted 而不是 AI-first**——把 AI 加进现有循环，但没有重新设计这个循环。他的判断很直接：

> I see teams claim AI-first while running the same sprint cycles, the same Jira boards, the same weekly standups, the same QA sign-offs. They added AI to the loop. **They didn't redesign the loop.**
> 问题其实更大一些。当 Agent 真正进入研发流程之后，团队面对的不再只是"它会不会写代码"，而是：

- 它能看到什么
- 它能理解什么
- 它能做什么
- 它怎么知道自己做得对不对
- 系统怎么防止它长期失控
- 人在这个过程中应该扮演什么角色

换句话说，真正发生变化的不是"多了一个工具"，而是——**我们的研发系统，开始需要对 Agent 友好**。

这也是 Harness Engineering 真正重要的地方。它不是 prompt 小技巧，也不是某个工具的使用心得，而是在讨论：当研发系统开始围绕 Agent 重构时，我们到底应该建设什么。

### （二）一个更准确的总纲：人类掌舵，Agent 执行（**Humans steer. Agents execute.**）

如果要把这件事压缩成一句话，我更愿意用 OpenAI 那篇文章里的表达：

> **人类掌舵，Agent 执行。**
> 这不是口号，而是一条系统分工原则。在传统团队里，很多价值主要来自工程师亲手完成实现：自己查代码、自己跑环境、自己改逻辑、自己验证结果、自己兜底隐性知识。在面向 Agent 的工程环境里，人的高价值工作会逐渐迁移到：定义目标、拆解任务、提供清晰参考、设计工程环境、设定边界和约束、判断风险、处理异常、沉淀规则。

与此同时，Agent 的价值也不在"替代工程师"，而在于承担越来越多**可定义、可执行、可验证、可迭代**的执行工作。

人和 Agent 不是简单替换关系，而是：

- 人负责方向、约束、判断和异常处理
- Agent 负责高吞吐执行、局部搜索、机械化验证和反馈驱动的迭代

如果一个团队看起来已经在"用 Agent"，但关键知识还藏在人脑里、流程经验还靠人工带着走、结果判断还要人逐项兜底，那么 Agent 实际上仍然只是在"辅助 coding"。所以 Harness Engineering 本质上是在重写这套分工：

> 让人少做隐式兜底，让系统多承接可读、可执行、可验证的工程能力。

### （三）为什么很多团队用了 Agent，却还是落不了地

这几年很多团队已经在用各种 coding agent，但实际效果差异很大——有些团队明显提速，有些团队只是多了一层热闹的自动化。这背后的原因，通常不在模型本身，而在工程系统本身。

一个很实用的判断标准是：

> Agent 最适合的任务，通常同时满足两个条件——**任务定义清晰**、**结果验证自动化**。
> 真正适合 Agent 发挥价值的任务不一定最复杂，但往往目标清楚、参考明确、边界明确、done 条件明确、验证路径明确、反馈速度足够快。

反过来，如果任务本身就很模糊，参考散落在聊天记录、会议纪要和资深工程师脑子里，验证又主要依赖人工判断，那么 Agent 很容易：

- 在错误方向上高速推进
- 重复尝试同类错误
- 做了很多动作，但没有稳定产出
- 把大量负担重新抛回给人工评审者

这时候更强的模型也未必能解决问题。所以很多 Agent 项目的问题不是模型不够强，而是系统没有为 Agent 提供足够好的任务表达、上下文结构、动作入口、反馈回路和长期约束。

> 当事情失败时，更好的反应不是"再试一次"，而是追问：系统到底缺了什么能力，为什么 Agent 看不到、做不到，或者做完之后不知道结果。

## 三、Agent 闭环与四层工程能力

### （一）Agent 不是一次生成，而是一个闭环系统

![](https://rte.weiyun.baidu.com/wiki/attach/image/api/imageDownloadAddress?attachId=ef6a103cd1ab4423b76ca71bd0774156&docGuid=_WYqybJ-OnKZXr)
要把 Agent 的工作方式讲清楚，一个很有用的框架是 **Perceive / Decide / Act / Feedback** 四个环节。

- **Perceive（先看见世界）**：Agent 需要先读取任务、代码、文档、历史决策、环境状态和已有结果。它的一切判断都依赖当前能看到什么。所以第一层问题不是"模型强不强"，而是系统有没有把关键上下文以 Agent 能消费的方式暴露出来。
- **Decide（判断下一步做什么）**：读完上下文之后，Agent 要判断先查什么、先改什么、先验证什么、什么时候该停、什么时候应该升级给人。在复杂任务里，它还要涉及任务拆解和推进顺序安排。
- **Act（真的采取动作）**：Agent 不只是输出建议，还要改代码、跑脚本、执行测试、调用工具、查询日志和指标、触发工作流、发起 PR、响应评论。只有能真正进入动作层，它才不是一个"会说怎么做"的模型，而是一个能参与工程闭环的执行体。
- **Feedback（根据结果继续修正）**：动作发生之后，Agent 还要知道测试有没有通过、脚本有没有报错、页面是不是正常、性能有没有退化、指标有没有变化、改动到底有没有生效。这些结果如果不能回流，下一轮决策就无从谈起。

一旦把 Agent 理解为 "感知 → 决策 → 行动 → 反馈 → 再感知" 的闭环系统，Harness Engineering 的工作就清楚了：

> 不是给它更多提示词，而是补齐这个闭环各环节所依赖的工程能力。

### （二） 从 Agent 闭环到工程落地：四层能力

把上面的闭环翻译成工程语言，我们真正要建设的是四层能力：

| Agent 运行闭环 | 对应工程能力        |
| -------------- | ------------------- |
| Perceive       | 让 Agent 读懂       |
| Decide + Act   | 让 Agent 可执行     |
| Feedback       | 让 Agent 有反馈闭环 |
| 长期稳定运行   | 让 Agent 可治理     |

这里最关键的区分是：反馈闭环关注"单次任务做完之后怎么知道结果"；可治理关注"整套系统长期能不能稳定、可控、不持续熵增"。把这两层分开，是为了不让反馈混进执行、不让治理混成一堆模糊流程要求。

### （三）第一层：让 Agent 读懂

这一层要回答的问题是：Agent 能看到什么、能理解什么、能信任什么。很多团队在引入 Agent 后最先遇到的问题，不是工具不够，而是 Agent 虽然能读仓库，但读不懂团队真正依赖的知识结构。

#### 1.代码仓库必须成为记录系统

如果一个团队真的想让 Agent 稳定参与工程工作，就必须接受这个前提：关键知识——架构约定、产品规格、设计决策、执行计划、技术债、待办事项、运行说明、质量约束——都必须进入代码仓库。

OpenAI 团队在五个月、约一百万行代码、1500 个 PR 的实战之后，把这件事概括成一句话：

> 对 Agent 来说，它在运行时无法在情境中访问的任何内容都是不存在的。
> 存储在 Google Docs、聊天记录或人们头脑中的知识，对它来说就等于不存在。这条原则看起来简单，但对很多团队的工程习惯冲击很大：Slack 里的讨论、飞书里的约定、会议里的共识、资深工程师脑子里的经验，如果没有进 repo、没有被版本化、没有在 Agent 的上下文里可见，那么 Agent 就会反复踩坑——因为这个约束并不在它能读到的世界里。

#### 2.文档不是越多越好，而是要可导航、可索引、可继续使用

"让 Agent 读懂"并不意味着堆更多文档。OpenAI 团队一开始尝试过"一个大型 AGENTS.md"的做法，结论是明确失败：

- 情境是稀缺资源，一个巨大的指令文件会挤掉任务、代码和相关文档
- Agent 要么错过关键约束，要么针对错误的约束优化
- 过多的指导反而变得无效
- 一旦人停止维护，它会悄然成为陈旧规则的坟场，Agent 无法判断哪些信息仍然有效

他们的修正是把 `AGENTS.md` 缩减到大约 100 行，作为**内容目录 / 入口地图**，而把真正的知识库放到一个结构化的 `docs/` 目录里，当作记录系统来用：设计文档被编目和索引，计划被视为一流工件，活跃计划 / 已完成计划 / 已知技术债务都被版本化、集中存放。专职的 linter 和 CI 作业验证知识库的更新状况、交叉链接和结构；一个定期运行的 **doc-gardening Agent** 扫描不再反映真实代码行为的过时或废弃文档，发起修复 PR。

所以一个更理想的状态通常是：

- 简短的 `AGENTS.md` 作为入口地图
- 结构化的 `docs/`，明确分类，交叉链接
- 能区分 `active / completed / deprecated`
- 有 freshness / ownership 机制
- 有基础校验，避免关键文档失效

#### 3.复杂任务要工件化，而不是每轮重新解释背景

对复杂任务来说，仅靠一轮 prompt 很难稳定推进。更有效的方式是把任务相关知识变成可持续使用的工件：design doc、exec plan、decision log、progress notes、风险边界、done condition。

特别要强调：**规划本身也应该成为工件**。任务拆解不应该只存在于某一次对话里，也不应该只存在于某个人脑子里，它应该被显式写下来并进入版本控制，让后续的人和 Agent 都能沿着它推进。一个真正有用的执行计划通常包含：

- 任务拆解
- 当前进度
- 关键决策
- 边界条件
- 暂不处理项
- done 条件

在复杂任务里，planner 不是可有可无的附加能力，而是连接"理解目标"和"采取动作"的中间层。

> "让 Agent 读懂"真正的目标，不是多写几篇文档，而是把关键知识和关键计划一起变成 Agent 能消费、能定位、能信任、能继续引用的系统资产。

### （四）第二层：让 Agent 可执行

这一层要回答的问题是：Agent 即使看懂了，能不能真的做事？很多团队在这一层的误区是把"会建议怎么改"误当成"具备执行能力"。

#### 1.从"会建议"到"能行动"

一个真正能进入研发闭环的 Agent，通常至少要能：

- 运行本地脚本
- 调用 test / lint / typecheck
- 使用仓库内标准命令
- 调用工具链
- 操作 GitHub / PR 流程
- 查看和回复评论
- 推进改动直到完成

#### 2.执行入口必须标准化、仓库内可见、版本化

如果仓库里的真实工作方式是：

- "这个脚本其实在某个人电脑上"
- "这个命令你问一下老同事"
- "这个流程平时都是手动做"
- "这个检查得靠 reviewer 自己记住"

那么 Agent 很难形成稳定执行。真正对 Agent 友好的环境，是那些把命令、技能、约束、调用路径全部显式化在 repo 中的系统。

OpenAI Developers 那篇 _Using skills to accelerate OSS maintenance_ 的核心做法，就是用 repo-local skills、`AGENTS.md`、scripts 和 GitHub Actions，把 verification、integration testing、release preparation、PR review 这些 recurring work 组织成可重复、可路由、可复用的工作流。

#### 3.运行环境本身也属于 Harness

很多团队讨论 Harness 时只想到 prompt、脚本、CI，但 Agent 的执行能力不只是"改文件"，还包括"接触运行中的系统"。OpenAI 团队的做法很值得参考：

- 让应用可以基于 **git worktree** 启动，每次更改 Codex 都能启动并驱动一个独立实例
- 把 **Chrome DevTools Protocol** 接入 Agent 运行时，创建处理 DOM 快照、屏幕截图和导航的技能，让 Codex 可以复现 bug、验证修复并直接推理 UI 行为
- 日志、指标、trace 通过本地可观测堆栈展示给 Codex，Agent 可以用 **LogQL** 查询日志、**PromQL** 查询指标

有了这些，"确保服务启动在 800ms 内完成"或"这四个关键用户旅程中的任何 span 都不得超过两秒"这样的提示才变得可行。如果 Agent 只能改静态代码却不能接触真实运行时，它很多时候就只能停留在"建议层"。

#### 4.光有动作能力还不够，还要约束动作空间

让 Agent 可执行并不等于放开它"随便做"。越是高吞吐执行系统，越需要明确约束：

- 模块边界
- 依赖方向
- 跨层调用限制
- 命名规则
- 文件复杂度限制
- 输入输出边界统一
- observability / reliability / security 要求模板化

OpenAI 团队通过自定义 linter（由 Codex 自己生成）和结构测试机械地强制业务域分层 `Types → Config → Repo → Service → Runtime → UI` 和依赖方向，同时把修复指令直接写进 lint 的错误信息里，让 Agent 看到报错就知道怎么改。这类约束在以人为本的工作流程中可能显得迂腐，但对 Agent 来说是**倍增器**——一旦编码，就能立即应用于所有地方。

这里有一个重要的工程原则：

> 文档管不住的事，就要逐步升级成规则。

- 文档管不住 → 升级成 lint
- lint 管不住 → 升级成类型约束
- 类型系统管不住 → 升级成结构测试 / CI 门禁

所以这一层真正的目标不是给 Agent 更多自由，而是：

> 让它的决策能稳定落到工程动作上，同时让错误动作尽可能被系统拦住。

### （五）第三层：让 Agent 有反馈闭环

这一层回答的问题是：Agent 做完动作之后，如何快速知道结果，并进入下一轮，而不是靠猜？

这是最容易被低估、也最容易被欠账的一层。很多团队在引入 Agent 时的第一反应是给它接更多工具，让它"能做更多事"。但更常见的深层问题其实是：

> **Act 很富，Feedback 很穷。**
> Agent 能跑命令、能改代码、能发起很多动作，但看不到这些动作的真实后果，或者只能通过非常弱的方式看到结果。结果就是它会盲做、会在错误方向上快速推进、会重复尝试相同错误、会把大量判断重新推回给人工评审者。

#### 1.结果必须返回给 Agent，而不只是展示给人

要让 Agent 真正进入闭环，关键不是"结果存在"，而是"结果必须能返回给 Agent"：

- 工具执行结果
- 测试结果
- CI 报错
- benchmark 结果
- profiler 结果
- 页面验证结果
- 脚本输出
- done / not done 判定信号

如果这些只显示在界面上给人看，对 Agent 来说反馈层依然是缺失的。

#### 2.可观测性必须对 Agent 可消费

- 日志最好是结构化的
- 指标可查询
- trace 可查询
- 性能目标可验证
- 故障定位路径清楚

否则 Agent 虽然能做动作，但一遇到线上相关问题就只能"猜"。OpenAI 团队为了这点专门做了本地可观测堆栈，让 Codex 能用 LogQL / PromQL 主动查询——可观测性对 Agent 的意义，从"人读控制台"升级成了"Agent 可查询的 API"。

#### 3.done / not done 必须尽量机械化

反馈层最关键的一点，是把"完成没有"尽量从主观判断转成机械化判断：

- 自动化测试
- lint / typecheck
- 验证脚本
- 页面检查
- benchmark 阈值
- 规则校验
- 明确的 failure signal

如果一个任务是否完成最终只能靠 reviewer 说"看起来差不多"，Agent 的闭环能力就会被大幅削弱。

#### 4.闭环的目标是自验证、自修复、再验证

真正成熟的反馈层，不只是让 Agent "看到失败"，而是支持它：

1. 检查自己改动的结果
2. 读取具体失败原因
3. 根据反馈修复
4. 修复后再验证
5. 直到满足通过条件

OpenAI 团队描述的 Codex 最近跨过的门槛，就是端到端地完成这个闭环：

> 验证仓库当前状态 → 重现 bug → 录制演示视频 → 实施修复 → 运行应用验证 → 录制第二个视频 → 开 PR → 回应反馈 → 检测并修复构建故障 → 仅在需要判断时交由人工。
> 所以没有反馈闭环时，Agent 只是会行动；有了反馈闭环，Agent 才开始具备稳定迭代的能力。

### （四）第四层：让 Agent 可治理

这一层关注的是长期稳定运行。它要回答的不是"这次任务能不能跑通"，而是：

- 这套系统长期是否稳定
- 质量是否可控
- 是否会持续熵增
- 是否会不断复制低质量模式
- 吞吐上来后流程会不会失控
- 模型升级后系统能不能跟着演化

#### 1.先定义信任边界，而不是默认全自动

Agent 的自主级别不应该模糊存在。更稳的做法是显式定义：

- 什么任务可以自主完成
- 什么任务只能半自动
- 什么任务必须人工把关
- 遇到哪些情况必须升级给人
- 某类任务出问题后如何回退到更保守模式

可治理的前提不是大胆放权，而是清楚知道放权到哪一层。

#### 2.质量不能靠"看起来差不多"，要靠门禁和评审机制

长期运行时，质量管理必须越来越机械化：lint、CI、结构测试、规则检查、Agent review 处理常规检查，人工只聚焦高风险、高不确定性问题。OpenAI 团队把几乎所有审核都调整成了"Agent 对 Agent"的方式——人类可以审 PR，但并非必须。

#### 3.高吞吐时代，合并哲学会变化

当 Agent 明显提升产出吞吐后，传统的 merge 节奏会承受很大压力。系统往往会逐渐倾向于 PR 更小、周期更短、对偶发失败有明确策略、不再总是无限等待"最完美的合并前状态"。OpenAI 团队给出的原则是：

> 在一个 Agent 吞吐量远超人类注意力的系统中，**纠错成本低，而等待成本高**。在低吞吐环境中这样做是不负责任的，而在这里，这通常是正确的选择。
> 这里有个前提必须说清楚：**"纠错便宜，等待昂贵"不是起点，而是结果**。只有在前面几层能力已经比较稳时——读得懂、动得了、有反馈、有约束——这种高吞吐节奏才成立。否则它只会把错误更快地扩散出去。

#### 4.真正难的是长期熵治理

Agent 会复制仓库中已有的模式。如果仓库里有坏模式、低质量样板、过时结构，它们很容易被持续放大。

OpenAI 团队最开始是人工处理这件事的——每周五花 20% 的时间清理 "AI 残渣"，很快发现不可扩展。他们的修正是把"黄金原则"（主观的机械规则，用于保持可读性和一致性）直接编码进仓库，并建立循环清理流程：

- 一组后台 Codex 任务定期扫描偏差
- 更新质量等级
- 发起有针对性的重构 PR
- 大多数可以在一分钟内审完自动合并

他们把这个过程描述为**垃圾回收**：

> 技术债务就像一笔高息贷款，不断地以小额贷款的方式偿还，总比让债务累积再痛苦地一次解决要好得多。

#### 5.Harness 本身也要演化

Harness 不是一次搭完就结束的。模型能力会变化，任务类型会变化，系统规模会变化，风险边界也会变化。一个成熟的治理层还需要包含：

- Harness 关键设计的变更记录
- 为什么这样设计的说明
- 模型升级后的同步审视
- 对旧规则和旧流程的淘汰机制

否则旧 Harness 很容易反过来变成新模型能力释放的包袱。

> 这一层真正关注的，是如何让一套 agent-first 工程系统长期稳定、持续提速、同时不失控。

## 四、组织与推进

### （一）组织前提：真正变化的不只是工具，还有人的角色

Harness Engineering 不只是工程问题，也是组织问题。一旦 Agent 真正进入研发闭环，人的角色会发生几个迁移。

- **从直接实现者，变成任务定义者**：过去很多任务之所以能落地，是因为工程师自己心里知道目标、边界、拆法和完成标准。当 Agent 参与进来后，这些原本藏在人脑里的东西必须被显式化——把目标定义清楚、把边界写清楚、把 done condition 写清楚、把任务拆得足够可执行。
- **从手工执行者，变成环境设计者**：以前很多事情是"工程师知道怎么做，所以自己去做"。现在更重要的是把知识写进 repo、把流程接进系统、把规则固化到 lint / CI / 测试 / runbook、把经验变成 Agent 可使用的能力。人的工作越来越像——为 Agent 设计环境，而不是代替 Agent 干活。
- **从全量审核者，变成关键节点把关者**：继续让人逐行审查所有 Agent 生成的代码，吞吐很快会超过人类注意力上限。更合理的做法是：常规检查交给自动化和 Agent review，人工聚焦方向判断、架构边界、风险控制和异常升级。
- **从经验持有者，变成规则沉淀者**：传统团队里很多工程智慧存在于资深工程师脑子里，但 Agent 无法使用"隐性经验"。这些经验必须被转成文档、规则、lint、CI、checklist、runbook、结构测试、模板、skills。

从组织上看，Harness Engineering 的前提就是：

> 把原本由人隐式承担的工程智慧，逐步外化成 Agent 可读、可执行、可验证的系统资产。

### （二）团队应该如何判断自己最该先补哪一层

理解上面的结构之后，真正落地时不能直接跳到"全都做"。更务实的做法是先看现状。

**先看我们已经有什么基础**：

- repo 结构是否已经清晰
- docs 是否已有基本框架
- scripts 是否比较全
- test 是否可用
- CI 是否成熟
- observability 是否存在
- 基础规则是否已经机械化

**再看我们当前最大的短板在哪**：

- 关键知识是否还在人脑和聊天里
- Agent 是否读不懂真实上下文
- 动作入口是否不标准
- 运行时信息是否无法回流
- done condition 是否仍然模糊
- 规则是否仍主要靠 reviewer 记忆
- 治理是否还停留在口头约束

**最后看下一步最值得投入什么**：关键不是一次性做完整套体系，而是判断目前最限制 Agent 发挥的，到底是读不懂、做不了、缺反馈，还是缺治理。

这也是 checklist 有价值的地方。它不是为了打出一个漂亮分数，而是为了帮助团队定位：

- 当前瓶颈在哪
- 下一步最值得投什么
- 哪些问题属于组织前提
- 哪些问题属于工程建设

### （三）推进方法：按任务纵切，不按能力横切

这件事最容易走偏的地方，是试图先在纸面上设计一套"完整 Harness"，然后再让 Agent 真正干活。

不要：

> 先补文档 → 再补工具 → 再补治理 → 直到"感觉差不多"才开始让 Agent 跑真实任务。
> 而要：

1. 选一个真实、近期就要完成的任务
2. 让 Agent 尝试完整跑一遍
3. 看它具体卡在哪一环
4. 只补当前缺口
5. 把这一条闭环真正打通
6. 再换下一个任务

> 按任务纵切，不按能力横切。
> 这样做能避免团队陷入"体系建设先行、但迟迟不进真实场景"的空转，也能让系统建设始终围绕真实瓶颈，而不是围绕想象中的最佳实践。

### （四）一个更务实的优先级：先补清晰度和可验证性，再扩执行和治理

从大多数团队的起步经验看，最常见的第一短板并不是模型能力，而是：

- 任务不够清晰
- 参考不够集中
- done condition 不明确
- 验证不够自动化

所以团队起步时最值得先补的通常是：

- 清晰参考
- 架构蓝图
- design doc / exec plan
- done condition
- lint / test / typecheck
- 验证脚本
- 明确的反馈信号

等真实任务开始卡在动作入口和运行时可见性上，再继续补：

- 标准脚本入口
- worktree / sandbox
- 页面访问能力
- observability 接入
- PR / review / workflow 接入

当吞吐真的起来之后，再强化：

- 架构约束
- 质量门禁
- Agent review
- 熵治理
- merge 策略调整
- Harness 演化治理

> 最稳的推进方式从来都不是一步到位，而是：先补最影响闭环质量的短板，再逐步扩大 Agent 的动作空间和系统治理能力。

## 五、结论：Harness Engineering 的本质

如果把全文再压缩成一句话：

> 当我们开始认真讨论 Harness Engineering 落地时，真正的问题已经不是 "Agent 会不会写代码"，而是 "我们的工程系统，是否已经准备好让 Agent 稳定地读、做、验证、迭代，并在长期运行中保持可控"。
> 从这个角度看，Harness Engineering 真正要回答的不是一个概念定义题，而是四个很具体的问题：

- 如何让 Agent 读懂
- 如何让 Agent 可执行
- 如何让 Agent 形成反馈闭环
- 如何让 Agent 长期可治理

而这背后对应的，也不只是工具接入，更是：

- 研发系统的重构
- 工程规则的机械化
- 运行时反馈的回流
- 组织分工的迁移
- 长期质量和熵的治理

所以 Harness Engineering 不是"给 Agent 加一层壳"，而是：

> 把研发系统改造成一个适合 Agent 稳定参与的工程环境。
> OpenAI 团队用五个月、一百万行代码、1500 个 PR 验证了这条路径在真实工程里可以走通。他们最后留下的那句话值得放在文档结尾：

> 构建软件仍然需要纪律，但纪律更多地体现在支撑结构上，而不是代码上。保持代码库一致性的工具、抽象和反馈回路变得越发重要。
> 这也是我们自己应该投入的方向。

## 附录 A：最值得反复使用的几个判断

下面这些判断建议在内部分享或后续讨论中反复使用，因为它们能帮助团队快速建立统一心智模型。

### A.1 失败时不要"再试一次"，而要问缺了什么能力

当事情失败时，更好的反应不是"再试一次"，而是追问：

- 我们到底缺了什么能力
- 为什么 Agent 看不到这件事
- 为什么它做不到
- 为什么它做完之后不知道结果
- 这件事应该如何变得对 Agent 可读、可执行、可验证

### A.2 对 Agent 来说，看不到的东西就等于不存在

Slack 里的讨论、飞书里的约定、会议里的共识，以及资深工程师脑子里的经验，如果没有进代码仓库、没有被版本化、没有在 Agent 的上下文里可见，那么对 Agent 来说就是不存在。

### A.3 文档管不住的事，就升级成代码

当某个规则反复被违反时，不要继续往文档里补一句"请注意"。更有效的做法是沿着这条路径升级：

- 文档管不住 → 升级成 lint
- lint 管不住 → 升级成类型约束
- 类型系统管不住 → 升级成结构测试 / CI 门禁

### A.4 Feedback 通常比 Act 更容易被欠账

很多团队第一反应是接更多工具，让 Agent 能"做更多事"。但更常见的问题是：

- Act 很富
- Feedback 很穷

也就是：它能动，但看不到后果。而这往往才是 Agent 看起来"不聪明"的深层原因。

### A.5 纠错便宜，等待昂贵——但有前提

当 Agent 吞吐量远高于人类注意力时，很多纯人工时代的流程会反向成为瓶颈。这时"快速合并 + 持续观察 + 快速修正"有可能比"无限期等待最完美状态"更合理。但这**不是起点，而是结果**。只有在前面几层能力已经比较稳时，这种节奏才成立。

## 附录 B：一些很常见的反模式

下面这些情况看起来像"已经建设了能力"，但很多时候只是表面成立。

### B.1 有 `AGENTS.md`，但它其实是一份大全

如果 `AGENTS.md` 变成一份巨大的操作手册，什么都往里堆，最后往往会变得：

- 不好导航
- 重点不突出
- 容易过期
- 对 Agent 也不友好

更好的做法是让它做入口地图，而不是做一切内容的容器。OpenAI 团队的经验是把它压到 100 行左右，只做目录与指路。

### B.2 有 `docs/`，但没有 freshness 和状态管理

文档不是"存在"就够了。如果没有 `active / completed / deprecated` 区分、没有 ownership、没有 freshness 机制，很多文档最终会变成噪音。

### B.3 能跑测试，但错误只对人可读

这类情况表面上看有反馈，实际上对 Agent 来说仍然没有形成有效闭环——它看到的只是"失败了"，但读不到有用的错误上下文。把修复指令写进 lint 错误信息、把测试失败的结构化上下文回流给 Agent，才算真正闭环。

### B.4 有 observability，但 Agent 查不到

很多团队已经有日志、指标、trace。但如果这些能力只能由人通过控制台人工读取，而不能进入 Agent 的工作流，那它们对 Agent 的价值仍然很有限。让 Agent 能用 LogQL / PromQL 主动查询，是一个很值得参考的方向。

### B.5 有规则，但没机械化

如果一个规则仍然主要靠 reviewer 记忆，那它本质上还是隐性经验，而不是系统能力。

### B.6 有治理流程，但吞吐一上来就崩

有些流程在低吞吐时看起来很稳，一旦 Agent 真正提升产出，立刻就会暴露瓶颈：

- PR 堆积
- reviewer 过载
- 偶发失败无人处理
- merge 节奏失控

这往往说明治理层还没有为高吞吐环境做好准备。
